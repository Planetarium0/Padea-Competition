# Automated Caterer Switching

Programmatic detection of underperforming caterers and a coordinator-gated
switch to a replacement — based on rolling `Caterer Feedback` ratings.

**Implementation status:** complete. See `scripts/actions/evaluate_caterers.py`
and `scripts/actions/execute_caterer_switch.py`. Webapp changes are documented
in `plans/new/caterer-switching-webapp.md`.

---

## Problem statement

Caterer quality degrades over time. Today there is no signal that triggers a
review. The `Caterers.Able to Serve Schools` field already encodes available
replacements, and `register_orders.py` already fetches `Caterer Feedback` —
both were unused.

---

## Trigger condition

Evaluate a caterer for a switch when **all** of the following hold:

| Constant | Value | Meaning |
|---|---|---|
| `SWITCH_THRESHOLD` | 2.5 | rolling average at or below this → propose switch |
| `WATCH_THRESHOLD` | 3.0 | rolling average at or below this (but > SWITCH) → warn only |
| `MIN_SESSIONS` | 3 | minimum distinct sessions with feedback in window |
| `MIN_RATERS` | 5 | minimum unique students rating in window |
| `ROLLING_WINDOW` | 4 | most-recent N sessions to average |

Evaluation is per **(school, caterer) pair** — a caterer can perform well
at one school and poorly at another.

Only the **current** caterer for a school is eligible for a switch proposal
(stale pairs from prior terms are logged but skipped).

---

## Candidate selection

1. Find all caterers where the school appears in `Able to Serve Schools`.
2. **Hard dietary filter**: a candidate is eligible only if it has at least
   one compatible menu item for *every* non-opted-out student at the school.
   If even one student cannot be covered, the caterer is excluded entirely.
   No manual accommodations are assumed.
3. Score remaining candidates:
   - `score = 0.6 × avg_rating_at_this_school + 0.4 × avg_rating_overall`
   - If the candidate has no history at this school, `score = avg_overall`.
   - Default overall average: 3.0 (neutral) when no history exists.
4. Top-scoring candidate becomes the proposal. If no candidate passes the
   dietary filter, a "no replacement available" alert email is queued
   (no proposal record is written).

## No-repeat suppression (term-scoped)

`evaluate_caterers.py` checks existing `Caterer Switch Proposals` before
creating a new one for a given (school, caterer) pair:

- **Active proposal exists** (Status = Pending / Approved / Executed) → skip silently.
- **Rejected proposal this term** → skip with log message.
  "This term" = on or after the start of the current Queensland school term
  (hardcoded 2026 dates in `QLD_TERM_STARTS`; update annually).

This means a coordinator who rejects a proposal won't be emailed about the
same caterer again for the rest of that term.

## Timing

Runs after `register_orders.py` (Wed 8:00 PM):

| Time | Command |
|---|---|
| Wed 8:00 PM | `./run orders generate` |
| Wed 8:05 PM | `./run evaluate` |
| Thu 3:00 PM | `./run orders send` |

This week's orders are already locked. The proposal's `Effective Week` is set
to **Monday of the week after next** — the earliest week the switch can affect.

If the coordinator approves and runs `./run switch <id>` before Thursday
3 PM, this week's `send_orders.py` picks up the new caterer automatically
(because `register_orders.py` already ran and the order was generated with
the new caterer after the flip).

Wait — actually the order for next week was already generated by
`register_orders.py` at 8 PM *before* the switch executes. So the switch
affects the *following* week's order, not next week's. `Effective Week` is
set accordingly.

## Data changes on execution

`execute_caterer_switch.py <proposal_record_id>` performs these changes:

| Table | Field | Change |
|---|---|---|
| `Sessions` | `Incoming Caterer` | Set to incoming caterer for all sessions at the school |
| `Caterers` (outgoing) | `Serves Schools` | Remove school |
| `Caterers` (outgoing) | `Able to Serve Schools` | Add school (so they remain a future candidate) |
| `Caterers` (incoming) | `Serves Schools` | Add school |
| `Caterers` (incoming) | `Able to Serve Schools` | Remove school |
| `Students` | `Meal Preference` | Cleared for all students enrolled at the school |
| `Caterer Switch Proposals` | `Status` | → `Executed` |

`Sessions.Caterer` is **not** changed immediately. The actual flip
(`Caterer ← Incoming Caterer`, `Incoming Caterer ← []`) happens at the
start of the next `register_orders.py` run (`flip_incoming_caterers()`).
This keeps the session in a transition state so the webapp can show the
incoming menu for preferences while attributing ratings correctly to the
current caterer.

## New Airtable table: `Caterer Switch Proposals`

Defined in `data/schema.py`. Fields:

| Field | Type | Purpose |
|---|---|---|
| Proposal ID | singleLineText | Primary key (human-readable, auto-generated) |
| School | multipleRecordLinks → Schools | Affected school |
| Outgoing Caterer | multipleRecordLinks → Caterers | Caterer being replaced |
| Incoming Caterer | multipleRecordLinks → Caterers | Proposed replacement |
| Avg Rating | number | Rolling average that triggered the proposal |
| Sessions Sampled | number | How many sessions in the window |
| Unique Raters | number | How many distinct students rated in the window |
| Proposed On | date | Date the proposal was created |
| Effective Week | date | Monday of the first week the switch affects orders |
| Status | singleSelect | Pending / Approved / Rejected / Executed |
| Notes | multilineText | Coordinator notes |

---

## Schema additions

Beyond the new table, two fields were added:

- **`Sessions.Incoming Caterer`** — link to Caterers; set by execute,
  cleared by register_orders.
- **`Caterer Feedback.Session Date`** — ISO date set by the webapp when
  feedback is submitted. Used by evaluate_caterers for accurate rolling-window
  ordering. Falls back to `Sessions.Date` (representative migration date) for
  legacy records.
- **`Scheduled Emails.Caterer Switch Proposal`** — optional link to
  `Caterer Switch Proposals` (alongside the existing optional `Weekly Order`
  link). Airtable doesn't support polymorphic links; both are optional fields.

## `./run` commands

```bash
./run evaluate [--dry-run]        # check ratings, propose switches
./run switch <proposal_record_id> [--dry-run]  # execute an approved switch
```

---

## Environment variable

```
PADEA_COORDINATOR_EMAIL=...    # address that receives proposals and alerts
```

Add to `.env`. If absent, proposals are still written to Airtable but no
email is queued.

## Edge cases

| Case | Handling |
|---|---|
| No eligible replacement | "No replacement" alert email; no proposal record |
| Active proposal already open | Skipped silently |
| Rejected proposal this term | Skipped with log message |
| Caterer no longer serving school | Feedback pair skipped as historical |
| Fewer than MIN_SESSIONS sessions | Skipped — not enough data |
| Fewer than MIN_RATERS unique raters | Switch suppressed; watch email if below WATCH_THRESHOLD |
| Student with no compatible item on incoming menu | Candidate excluded (hard filter, not a warning) |
| Order for next week already generated | Switch affects the following week (Effective Week set accordingly) |
| Dry-run mode | All write operations logged and skipped |
